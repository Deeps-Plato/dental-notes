  <behavior>
    State transitions:
    - SessionManager starts in IDLE state
    - start(mic_device, output_dir) transitions IDLE -> RECORDING
    - pause() transitions RECORDING -> PAUSED
    - resume() transitions PAUSED -> RECORDING
    - stop() transitions RECORDING|PAUSED -> STOPPING -> IDLE, returns transcript file path
    - Invalid transitions raise RuntimeError with clear message

    Recording behavior:
    - start() initializes AudioCapture, VadDetector, AudioChunker, WhisperService, TranscriptWriter
    - start() launches a background processing thread that: reads audio blocks from capture queue, feeds them to chunker, transcribes completed chunks via WhisperService, deduplicates via stitcher, appends to transcript writer
    - pause() stops the audio stream but keeps the processing thread alive
    - resume() restarts the audio stream
    - stop() stops audio stream, flushes remaining chunker buffer, transcribes final chunk if any, closes transcript writer, stops processing thread
    - Audio data is discarded after transcription (never stored as WAV) -- AUD-01

    Thread safety:
    - All state transitions protected by threading.Lock
    - Processing thread checks state to know when to stop

    Observability:
    - get_transcript() returns the accumulated transcript text so far
    - get_state() returns current SessionState
    - is_active() returns True if RECORDING or PAUSED
    - get_level() returns current audio level (delegates to AudioCapture)
  </behavior>
  <action>
    Create the session manager that orchestrates the full pipeline.

    1. Create `src/dental_notes/session/manager.py`:
       - Enum `SessionState` with values: IDLE, RECORDING, PAUSED, STOPPING
       - Class `SessionManager`:
         - `__init__(self, settings: Settings)`: Store settings, set state=IDLE, create lock
         - `start(self, mic_device: int | None = None) -> None`:
           - Acquire lock, verify state is IDLE
           - Create AudioCapture(settings), VadDetector(settings.vad_threshold), AudioChunker(settings, vad), TranscriptWriter(settings.storage_dir)
           - WhisperService is passed in or created (prefer injection for testability): `self._whisper` should be set before start() or created internally
           - Start audio capture
           - Launch `_processing_loop` in a daemon thread
           - Set state to RECORDING
         - `pause(self) -> None`: Lock, verify RECORDING, stop audio capture, set PAUSED
         - `resume(self) -> None`: Lock, verify PAUSED, restart audio capture, set RECORDING
         - `stop(self) -> Path`:
           - Lock, verify RECORDING or PAUSED
           - Set STOPPING
           - Stop audio capture
           - Flush chunker -- if remaining audio, transcribe it
           - Close transcript writer
           - Set _processing_thread_running = False, join thread
           - Set IDLE
           - Return transcript file path
         - `_processing_loop(self)`:
           - While _processing_thread_running:
             - If state is PAUSED: sleep briefly and continue
             - Get audio block from capture queue
             - If block is None: continue (queue empty)
             - Feed block to chunker
             - If chunker returns a completed chunk:
               - Transcribe chunk via whisper_service.transcribe(chunk)
               - Deduplicate overlap with previous transcript tail via deduplicate_overlap()
               - Append deduped text to transcript writer
               - Append to internal _transcript string (for get_transcript())
               - The audio chunk (numpy array) goes out of scope here -- DISCARDED, not stored
         - `get_transcript(self) -> str`: Return accumulated transcript
         - `get_state(self) -> SessionState`: Return current state
         - `is_active(self) -> bool`: Return state in (RECORDING, PAUSED)
         - `get_level(self) -> float`: Delegate to audio_capture.get_level(), return 0.0 if no capture active

       - IMPORTANT: Whisper inference MUST NOT run on the asyncio event loop. The _processing_loop runs in its own thread, which is correct. When the FastAPI routes (Plan 03) call session methods, those should be thread-safe (lock-protected).
