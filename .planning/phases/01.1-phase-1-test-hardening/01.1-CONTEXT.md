# Phase 1.1: Phase 1 Test Hardening - Context

**Gathered:** 2026-03-07
**Status:** Ready for planning
**Source:** User conversation — adopted TDD methodology after v1 delivered 128 tests and zero working product

<domain>
## Phase Boundary

This phase hardens Phase 1 (Streaming Capture & Transcription) by filling test coverage gaps, adding a pipeline integration test, and completing the human verification checkpoint. No new features are added — this is purely about proving Phase 1 actually works.

Phase 1 has 83 tests across 9 test files, but three production modules lack dedicated tests, no integration test exercises the full pipeline, and the human verification checkpoint (Plan 01-03 Task 3) was never completed.

</domain>

<decisions>
## Implementation Decisions

### Test Coverage Gaps (Locked)
- `audio/capture.py` has no dedicated test file — must be tested with realistic fakes (sounddevice is hardware-dependent)
- `ui/hotkey.py` has no dedicated test file — must be tested with pynput mocked (requires X display)
- `main.py` (app factory) has no dedicated test file — must test lifespan, router inclusion, static file mounting

### Integration Test (Locked)
- Must exercise the full Phase 1 pipeline: audio input → VAD → chunker → whisper service → stitcher → transcript writer
- Must use realistic fakes that behave like real components (not mocks returning canned strings)
- Must prove data flows through the chain and produces a transcript file on disk
- The integration test should catch the class of bugs that killed v1: "all units pass but the pipeline doesn't connect"

### Human Verification (Locked)
- Plan 01-03 Task 3 human verification checkpoint must be completed
- User must confirm the app works on real Windows hardware with real microphone
- This is a blocking gate — Phase 1 is not done until human says it works

### Claude's Discretion
- Specific test assertions and edge cases for capture.py, hotkey.py, main.py
- How to structure the integration test (single test function vs test class)
- Whether to use existing conftest fixtures or create new ones
- Test file naming follows existing pattern: `test_{module_name}.py`

</decisions>
