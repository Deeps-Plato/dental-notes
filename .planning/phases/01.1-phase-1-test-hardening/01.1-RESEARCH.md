**Researched:** 2026-03-07
**Domain:** Python test coverage, integration testing, hardware boundary testing
**Confidence:** HIGH

## Summary

Phase 1 has 83 passing tests across 9 test files covering 8 of 11 production modules. Three production modules lack dedicated tests: `audio/capture.py` (hardware audio capture), `ui/hotkey.py` (keyboard listener), and `main.py` (FastAPI app factory with lifespan). No integration test exercises the full pipeline from audio input through to transcript file on disk. The human verification checkpoint from Plan 01-03 remains incomplete.

The codebase has well-established testing patterns: hand-written fakes over mocks (FakeVadModel, FakeWhisperModel, FakeAudioCapture, FakeWhisperService, FakeChunker, FakeSessionManager), lazy imports for hardware-dependent modules (sounddevice, pynput, faster_whisper), and `unittest.mock.patch` for replacing model loaders. The integration test should reuse and compose these existing fakes to exercise the full pipeline: audio blocks -> VAD -> chunker -> whisper service -> stitcher -> transcript writer, proving data flows through and produces a transcript file with speaker labels.

**Primary recommendation:** Write three unit test files for the untested modules using boundary testing patterns (mock hardware interfaces, test logic), then write one pipeline integration test that composes existing fakes into a connected chain and asserts a transcript file is produced on disk with correct content.

### Pattern 1: Hardware Boundary Testing (for capture.py)
**What:** Test all logic around hardware interfaces by mocking only the hardware boundary (sounddevice), while testing the actual queue, callback, and level-meter logic.
**When to use:** `AudioCapture` wraps `sd.InputStream` (requires PortAudio). The callback, queue, `get_block()`, `get_level()`, and device listing functions contain testable logic.
**Key insight:** The `_audio_callback` method is a pure function that copies data and puts it in a queue. It can be called directly without a real stream. `list_input_devices()` and `find_device_by_name()` need `sd.query_devices()` mocked.

```python
# Pattern: Test callback directly without real PortAudio stream
from dental_notes.audio.capture import AudioCapture, BLOCK_SIZE
from dental_notes.config import Settings

capture = AudioCapture(Settings())
# Call callback directly -- no real stream needed
block = np.random.default_rng(0).random(BLOCK_SIZE).astype(np.float32)
capture._audio_callback(block.reshape(-1, 1), BLOCK_SIZE, None, sd.CallbackFlags())
assert capture.audio_q.qsize() == 1
retrieved = capture.get_block()
np.testing.assert_array_equal(retrieved, block)
```

### Pattern 2: Lazy Import Mocking (for hotkey.py)
**What:** `pynput` is imported inside methods, not at module level. Tests must mock at the import site or inject behavior.
**When to use:** `HotkeyListener` imports `pynput.keyboard.Key` and `pynput.keyboard.Listener` inside methods. On headless Linux (CI), pynput fails without X display.
**Key insight:** Test `_on_key_press` and `_process_actions` directly by mocking pynput imports and providing a fake session manager. The action queue pattern (key press -> queue -> worker thread) is highly testable.

```python
# Pattern: Test action processing without pynput
from unittest.mock import patch, MagicMock
from dental_notes.ui.hotkey import HotkeyListener

fake_mgr = FakeSessionManager()  # reuse from test_routes.py or test_session_manager.py
listener = HotkeyListener(fake_mgr)

# Directly test action processing
listener._running = True
listener._action_queue.put_nowait("toggle_start_stop")
# Call _process_actions in a controlled way (or test via start/stop lifecycle)
```

### Pattern 3: Lifespan Testing (for main.py)
**What:** Test `create_app()` factory produces a properly configured app, and lifespan sets up/tears down state correctly.
**When to use:** `main.py` creates the FastAPI app, mounts static files, includes router, and runs lifespan that initializes WhisperService, SessionManager, and HotkeyListener.
**Key insight:** The existing `test_routes.py` already bypasses lifespan by directly setting `app.state`. For main.py tests, use `httpx.AsyncClient` with the app's actual lifespan OR test `create_app()` structurally (router inclusion, static mount, title).

```python
# Pattern: Structural test of create_app()
from dental_notes.main import create_app

app = create_app()
assert app.title == "Dental Notes"
route_paths = [r.path for r in app.routes]
assert "/" in route_paths  # index route included
assert "/static" in [m.path for m in app.routes if hasattr(m, 'path') and 'static' in str(m.path)]
```

### Pattern 4: Pipeline Integration Test
**What:** Compose existing fakes into a connected pipeline and run data through the full chain.
**When to use:** This is the critical integration test that catches the class of bugs that killed v1.
**Key insight:** The `SessionManager` already orchestrates the full pipeline. The existing test_session_manager.py tests already use FakeAudioCapture, FakeWhisperService, and FakeChunker. The integration test should go further: use a FakeVadModel-backed real VadDetector, a real AudioChunker, a fake-model WhisperService, the real stitcher, and a real TranscriptWriter -- then assert the transcript file exists on disk with expected speaker-labeled content.

```python
# Pattern: Integration test using real components with fake hardware
# 1. Create real Settings with tmp_path
# 2. Create FakeAudioCapture with speech-like blocks
# 3. Create real VadDetector with FakeVadModel (speech probs)
# 4. Create real AudioChunker with the VadDetector
# 5. Create FakeWhisperService that returns dental text
# 6. Wire into SessionManager via factory overrides
# 7. Start session, wait for processing, stop
# 8. Assert transcript file exists on disk
# 9. Assert file contains speaker labels and dental text
# 10. Assert no audio data retained in memory
```

### Anti-Patterns to Avoid
- **Mocking the thing being tested:** Don't mock AudioCapture.get_block() in capture tests. Test the real queue logic.
- **Canned mock returns in integration test:** Don't use `MagicMock(return_value="text")`. Use realistic fakes that maintain state.
- **Testing pynput internals:** Don't test that pynput.Listener receives callbacks. Test that HotkeyListener routes F9/F8 to correct session actions.
- **Import-time failures:** All hardware imports (sounddevice, pynput, faster_whisper) are lazy. Tests must not break this pattern.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Async HTTP testing | Raw requests | `httpx.AsyncClient(transport=ASGITransport)` | Already used in test_routes.py; handles ASGI protocol correctly |
| Fake audio data | Random bytes | `np.zeros/random` with correct dtype=float32 | Existing conftest fixtures provide speech, silence, noise patterns |
| VAD in tests | Real silero model | `FakeVadModel` from conftest | Avoids 100MB model download; controlled probability sequences |
| Whisper in tests | Real faster-whisper | `FakeWhisperModel` / `FakeWhisperService` | Avoids GPU dependency; returns configurable text |
| Session state faking | Manual state tracking | `FakeSessionManager` from test_routes.py | Already mimics full state machine |

**Key insight:** The project already has excellent test doubles. The integration test should compose real components where possible (VadDetector with FakeVadModel, real AudioChunker, real TranscriptWriter) and fake only hardware boundaries (AudioCapture, WhisperModel).

## Common Pitfalls

### Pitfall 1: pynput Requires X Display on Linux
**What goes wrong:** `from pynput.keyboard import Listener` crashes on headless Linux (CI, WSL without X) with `Xlib.error.DisplayNameError`.
**Why it happens:** pynput's keyboard backend tries to connect to X11 at import time.
**How to avoid:** Mock `pynput.keyboard` before importing HotkeyListener methods that use it. The production code already uses lazy imports inside methods -- tests must respect this by patching the import path.
**Warning signs:** Tests pass locally with desktop but fail in CI.

### Pitfall 2: sounddevice Requires PortAudio
**What goes wrong:** `sd.InputStream()` fails without PortAudio library installed; `sd.query_devices()` returns empty or errors.
**Why it happens:** sounddevice wraps libportaudio which needs audio hardware drivers.
**How to avoid:** Never create a real `sd.InputStream` in tests. Test `_audio_callback` by calling it directly. Mock `sd.query_devices()` for `list_input_devices()` / `find_device_by_name()` tests.
**Warning signs:** `OSError: PortAudio library not found` in CI.

### Pitfall 3: Threading Timing in Integration Tests
**What goes wrong:** Integration test starts SessionManager, but processing thread doesn't consume all blocks before stop() is called.
**Why it happens:** The processing loop runs in a daemon thread. `time.sleep(0.5)` may not be enough, or may be flaky.
**How to avoid:** Use `FakeAudioCapture` with a finite block list (it returns None after all blocks are consumed). After `time.sleep()`, check that the expected number of transcribe calls occurred before asserting. The existing `test_session_manager.py` uses `time.sleep(0.5)` successfully -- follow the same pattern.
**Warning signs:** Flaky tests that sometimes pass, sometimes fail depending on system load.

### Pitfall 4: VadDetector Constructor Loads Model
**What goes wrong:** `VadDetector.__init__` calls `self._load_model()` which tries to download/load silero-vad.
**Why it happens:** The constructor eagerly loads the model.
**How to avoid:** Use `unittest.mock.patch.object(VadDetector, "_load_model", return_value=FakeVadModel(...))` -- exactly as done in test_vad.py and test_chunker.py.
**Warning signs:** Tests hang downloading silero-vad model, or fail with torch import errors.

### Pitfall 5: Lifespan Not Invoked in create_app() Tests
**What goes wrong:** Testing `create_app()` doesn't trigger lifespan, so `app.state.session_manager` is not set.
**Why it happens:** FastAPI lifespan only runs when the app is served (via ASGI), not when the app object is created.
**How to avoid:** For structural tests (router inclusion, static mount), test the app object directly. For lifespan tests, use `async with LifespanManager(app)` pattern (from `asgi-lifespan` library) or manually set `app.state` like test_routes.py does. Since we want to avoid adding dependencies, testing structural properties of `create_app()` is sufficient -- lifespan behavior is already covered by test_routes.py indirectly.
**Warning signs:** `AttributeError: 'State' object has no attribute 'session_manager'`.

### Pitfall 6: CallbackFlags Mock
**What goes wrong:** `_audio_callback` receives `sd.CallbackFlags` as 4th argument. Creating one in tests requires sounddevice.
**Why it happens:** `sd.CallbackFlags` is part of the sounddevice library.
**How to avoid:** The callback ignores `status` (the flags parameter). Pass `None` or a simple mock object. The method only uses `indata`, `frames`, `time_info` is `object`, and `status` is unused.

def test_audio_callback_drops_when_queue_full():
    capture = AudioCapture(Settings())
    block = np.zeros(BLOCK_SIZE, dtype=np.float32).reshape(-1, 1)
    # Fill queue to capacity (maxsize=200)
    for _ in range(200):
        capture._audio_callback(block, BLOCK_SIZE, None, None)
    assert capture.audio_q.qsize() == 200
    # 201st block should be silently dropped
    capture._audio_callback(block, BLOCK_SIZE, None, None)
    assert capture.audio_q.qsize() == 200  # Still 200, not 201

import time
import threading
from dental_notes.ui.hotkey import HotkeyListener
from dental_notes.session.manager import SessionState

def test_toggle_start_stop_starts_idle_session():
    fake_mgr = FakeSessionManager()  # from conftest or inline
    listener = HotkeyListener(fake_mgr)
    listener._running = True

    # Start worker thread manually (without pynput Listener)
    worker = threading.Thread(target=listener._process_actions, daemon=True)
    worker.start()

    time.sleep(0.3)

    listener._running = False
    worker.join(timeout=1.0)

    assert fake_mgr.get_state() == SessionState.RECORDING

### Example 5: Pipeline Integration Test Structure

```python
# Source: composition of existing fakes from test_session_manager.py + conftest.py
import time
from pathlib import Path
from unittest.mock import patch
import numpy as np
from tests.conftest import FakeVadModel
from dental_notes.config import Settings
from dental_notes.session.manager import SessionManager

def test_full_pipeline_produces_transcript_file(tmp_path):
    """Integration: audio blocks -> VAD -> chunker -> whisper -> stitcher -> file."""
    settings = Settings(storage_dir=tmp_path / "transcripts")

    # Fake audio: 9 blocks of speech-like data
    blocks = [np.random.default_rng(i).random(1600).astype(np.float32) for i in range(9)]

    # Fake whisper that returns dental text
    whisper = FakeWhisperService(responses=[
        "The patient presents with pain on tooth 14",
        "I recommend a crown preparation D2740",
        "Okay doctor sounds good",
    ])

    mgr = SessionManager(settings)
    mgr._whisper = whisper
    mgr._create_capture = lambda: FakeAudioCapture(blocks)

    # Use real VadDetector with FakeVadModel (all speech)
    speech_probs = [0.9] * 500
    fake_model = FakeVadModel(probabilities=speech_probs)
    with patch.object(
        __import__('dental_notes.audio.vad', fromlist=['VadDetector']).VadDetector,
        "_load_model",
        return_value=fake_model,
    ):
        mgr._create_chunker = mgr._default_create_chunker
        mgr.start()
        time.sleep(1.0)
        transcript_path = mgr.stop()

    # Assert transcript file exists and has content
    assert transcript_path.exists()
    content = transcript_path.read_text()
    assert len(content) > 0
    # Assert speaker labels present
    assert "Doctor:" in content or "Patient:" in content
```
